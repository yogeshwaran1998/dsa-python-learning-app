# Theory File Template & Authoring Guide

Every topic's `*.md` theory file should follow this structure so the app renders
consistently and learners get the same depth across topics. Keep prose tight and
interview-focused. Source files live at the repo root; `sync-content.mjs` mirrors
them into the app.

## Required section order

```markdown
# <Topic> - Theory Guide

## Table of Contents
1. [Section](#section)   <!-- anchors are auto-generated by rehype-slug -->
...

---

## Overview
One-paragraph definition: what it is and the core idea (1-3 sentences).

## When to Use
Bulleted signals that this technique/structure applies. Contrast with alternatives.

## How It Works
The mechanics, with a **styled diagram** (see below) or a short **worked example**.

## Complexity
| Operation | Time | Space |
|-----------|------|-------|
...
Note any amortized / best-vs-worst distinctions below the table.

## Core Patterns / Templates
Named patterns with a 1-2 line "when" + a short code snippet. Reference the
topic's `.py` file by function name (the runnable code lives there).

## Common Pitfalls
Edge cases and mistakes (empty input, off-by-one, overflow, duplicates, etc.).

## Interview Tips
Clarifying questions, optimization follow-ups, and what interviewers look for.

## Practice Problems
- **Easy:** ...
- **Medium:** ...
- **Hard:** ...

## Summary
A compact recap table (pattern → use case → time → space) or bullet cheat-sheet.
```

## Authoring rules

- **Tag every code block with a language** (` ```python `, ` ```text `). Untagged
  multi-line blocks now render as plain monospace, but tag them anyway for clarity
  and highlighting. Use ` ```text ` for example input/output and trace blocks.
- **Headings** drive the in-page Table of Contents (h2/h3). Use clear, unique
  heading text; ids are generated automatically.
- **Complexity tables** must be present for every data structure / algorithm.
- **Reference the `.py`**, don't duplicate large code — point to the function that
  implements each pattern (e.g. "see `two_sum()` in the code panel").

## Styled diagrams

Author diagrams as raw HTML using the semantic classes defined in
`dsa-learning-app/src/index.css` (they are theme-aware and survive Tailwind's
build). Prefer these over ASCII art for structural visuals.

Building blocks:
- `.dsa-diagram` — figure wrapper (add `--col`, `--center` as needed); end with an
  optional `<div class="dsa-caption">…</div>`.
- `.dnode` containing `.dcell` (cells); `.dcell--label`, `.dcell--muted` modifiers;
  node variants `.dnode--null`, `.dnode--green`, `.dnode--highlight`.
- `.darrow` (`--muted`) — connector between nodes.
- `.dindex` with `<small>` (label) + `<b>` (value) — an indexed array slot.

Example — a singly linked list node pointing to the next:

```markdown
<div class="dsa-diagram dsa-diagram--center">
  <div class="dnode">
    <span class="dcell">5</span>
    <span class="dcell dcell--label">next</span>
  </div>
  <span class="darrow">&rarr;</span>
  <div class="dnode">
    <span class="dcell">9</span>
    <span class="dcell dcell--label">next</span>
  </div>
  <span class="darrow">&rarr;</span>
  <div class="dnode dnode--null"><span class="dcell">null</span></div>
  <div class="dsa-caption">Singly linked list: each node stores a value and a pointer to the next.</div>
</div>
```

For trees/graphs, use inline `<svg>` with `stroke="currentColor"` / `fill` set to
theme colors (or the same classes) so they adapt to light/dark mode. ASCII fences
(` ```text `) remain a valid fallback for quick diagrams.
